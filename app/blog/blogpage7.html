Git With It

View my GitHub

I decided to set myself a project with the aim to develop my final year project in GitHub to illustrate the progress of change from start to finish. The objectives are to research Git and GitHub, to illustrate the knowledge by uploading the various code stages to GitHub through commits and possibly forking and to evaluate the use of Git and GitHub as a new user.

All about Git ……

Git was born in 2005 and was created by Linus Torvalds who also created Linux. It was first developed to manage the Linux development community. Some of the goals of the new system were (git, no date):

  • Speed
  • Simple design
  • Strong support for non-linear development (thousands of parallel branches)
  • Fully distributed
  • Able to handle large projects like the Linux kernel efficiently (speed and data size)

GitHub is a Git repository hosting service but has many other features of its own. It provides a Web-based graphical interface and provides access control and collaboration features for projects. GitHub also allows the use of “forking” in which someone can have their own copy of a repository to make changes and then those changes can be shared with the original owner by a pull request and merged if the original owner is happy with the results.

Why I wanted to use Git...

I decided that Git would be an ideal self-initiated project as version control is important when projects start getting larger,

“No matter how hard you try, when multiple people are working on a single project without a version control system in place things get chaotic” (Web Design Depot, 2009:online)

Even though I am currently only working on projects by myself I felt it was important to make sure I knew all the basics so if I ever get the chance to collaborate with others on a project then this is something that might be used or something I could introduce! The most useful aspect to version control is the fact that you have many different versions, when you save something and make a change the original copy isn’t lost.

Git is a distributed version control, this means (git,no date) that each copy of the file is a full copy and not just a snapshot. It also allows 2 people to work simultaneously on the same project and when it is pushed back up it will sort out who did what changes and merge them into the final copy, if the two people have changed the same thing Git will show it as conflict.

Set up

I wanted to learn how to use terminal to get my files onto GitHub and look at GitHub Desktop. I have previously looked at Git Bash on PC and have already created a GitHub account. I am now a Mac owner so will be using terminal.

How

To put my final year project up onto GitHub I first decided to get a good foundation. I did the Code School (Code School, No Date) tutorial called Git Real which gave me a brief look into all the different things I could do, Git Real also includes challenges at the end which really helped me solidify what I had learnt. The challenges are pretend situations and I found this useful as it made you think what you had learnt and apply it to a new situation.

After completing this I also watched the Lynda.com video called up and running with Git and Github (Lynda.com, 2015), this helped me set it up for my Mac. I found this useful as it allowed me to play the video and then pause it to do the same on my laptop, this was very helpful as I could then follow along.

The set up using terminal

I then decided to apply what I had learnt to my own work. My “go to” resource was the git website as this did step by step processes. I used the terminal on my Mac to set up my repository, add the files and commit them. Firstly I initialised Git in the folder by changing the directory ( cd ) then dragging the folder into terminal. Then git init initialises git within that folder and therefore watches for changes.

The files then need to be added using git add filename or git add . (the space after add followed by a full stop adds all files so can be quicker!) I then committed the files with a message which allows me to see what I have changed or improved, git commit -m "add a message".
git log allows you to see any previous messages.

Over to GitHub, I created a new repository and then two options are given which can simply be copied into terminal. I found this part quick and easy.

Once I got my final year project on GitHub I threw a small party, actually it was nowhere near as bad as I thought it would be! I think the terminal looks scary and when you see people typing lots of commands things can get complex but for the simple purpose of putting things on GitHub I found it straightforward!

Forking

Forking is being able to have your own copy of someone else’s code to work on and look at. If you make changes to it that benefit or improve the original, it is possible to send it back to the original person and they can choose to view the changes and merge them if they like. This then becomes a pull request. I did this for my friends project, she had some problems with id's and classes so I forked her repo, made the changes then created a pull request, she could then accept the changes.

“A fork is a copy of a repository. Forking a repository allows you to freely experiment with changes without affecting the original project.” (GitHub Help, No Date)

GitHub Desktop

This is when I decided to try GitHub Desktop (GitHub Desktop, No Date) even though I had not seen many people use this before I thought I would give it a go. I found it extremely easy and straightforward, it was perfect for keeping track of changes within the files and it showed me what had been updated and allowed me to commit to the master branch and then sync so that it updated my files on GitHub. It gives a more visual feel, I liked that but am happy using terminal.

Evaluation

This has been a very useful self-initiated project, I think Git and GitHub will come in very handy during projects in the future and I am going to continue putting my current projects on there. Both terminal and GitHub Desktop have been useful and I think both of them can be as equally as useful but more features are available in terminal. Becoming familiar with terminal is a good thing as it is used for many other things such as Gulp. I have been able to successfully upload my final year project to GitHub and keep it updated.
